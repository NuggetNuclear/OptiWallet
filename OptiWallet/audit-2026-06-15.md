# OptiWallet — Code Audit & Security Review

**Fecha:** 2026-06-15 · **Versión:** 0.1.0-beta · **Modo:** Auditoría (hallazgos, sin fixes aplicados)
**Alcance:** `app/`, `components/`, `lib/`, `scripts/`, `public/sw.js`, `proxy.ts`, configs, dependencias, docs.

## Resumen ejecutivo

El núcleo público (API de solo lectura) está **sólido**: queries 100% parametrizadas, validación de IDs, escape de comodines LIKE, security headers, sin `dangerouslySetInnerHTML`/`eval`, errores 500 genéricos. `tsc --noEmit` limpio, **138/138 tests pasan**, `npm audit` solo reporta 2 vulnerabilidades transitivas de **dev** (build-time).

El hallazgo central es de **proceso, no de un exploit puntual**: **las dos auditorías previas (`audit-report.md` 2026-06-10, `security-audit-2026-06-11.md`) son anteriores al panel de administración.** La de 2026-06-11 describe explícitamente la app como *"API pública de solo lectura, sin autenticación"*. Todo el sistema admin (`lib/admin-*`, `app/api/admin/*`, `proxy.ts` auth guard, sesiones HMAC, TOTP) **nunca fue auditado** — y `docs/SECURITY.md` enlaza esos reportes obsoletos como la referencia de seguridad.

No encontré vulnerabilidades críticas (sin bypass de auth, sin inyección, sin fuga de secretos). Sí hay 3 hallazgos Medium en la superficie admin y varias inconsistencias doc↔código que podrían inducir un "fix" que rompa la auth.

## Seguridad — Panel de administración (NUEVO, sin auditar antes)

### 🟠 M1 — El Service Worker cachea respuestas admin a pesar de `Cache-Control: no-store`

`public/sw.js` se registra en scope `/` (`lib/hooks/use-service-worker.ts`) → controla **todo** el origin, incluido `/admin` y `/api/admin/*`. El handler `fetch` aplica `networkFirstStrategy` a cualquier GET same-origin que matchee `isAPIRoute` (`/api/`), y cachea **toda respuesta `ok`** ignorando el header:

```js
if (networkResponse.ok) {
  cache.put(request, networkResponse.clone());   // ignora Cache-Control: no-store
}
```

Los route handlers admin devuelven `no-store` justamente para no persistir, pero el SW lo sobreescribe. Resultado: tras navegar el panel en producción, el `CacheStorage` del browser (`optiwallet-api-v2`) contiene la **lista de admins** (emails, `totp_enabled`, timestamps de login) y el **audit log de 30 días** (acciones, IPs). Persiste tras logout y tras expirar la sesión. En una máquina compartida es leíble desde DevTools → Application → Cache Storage.

**Fix:** excluir admin del SW al inicio del handler `fetch`:

```js
if (url.pathname.startsWith("/api/admin/") || url.pathname.startsWith("/admin")) return;
```

### 🟠 M2 — Bypass de rate limiting vía `X-Forwarded-For` spoofeable

`lib/admin-guard.ts`:

```ts
export function clientIp(req: NextRequest): string {
  return req.headers.get("x-forwarded-for")?.split(",")[0]?.trim() ?? "unknown";
}
```

Toma el **primer** valor de `X-Forwarded-For`, que es el extremo controlable por el cliente: si el atacante envía `X-Forwarded-For: <aleatorio>`, los proxies suelen *anteponer*/conservar ese valor. Rotando ese campo en cada request, el contador por IP de `admin_login_attempts` nunca llega a 5 → el throttle de 5/15min sobre `login` y `verify-totp` queda anulado, habilitando fuerza bruta de password y del código TOTP de 6 dígitos.

**Fix:** usar una fuente de IP confiable detrás de Vercel — `x-real-ip`, o el valor de IP que provee el runtime, en vez del primer hop de XFF. Verificar el comportamiento exacto de XFF en tu config de Vercel antes de elegir el índice.

### 🟠 M3 — Modelo de admin plano: takeover lateral entre admins

No hay roles ni super-admin. En `app/api/admin/users/[id]/route.ts`, cualquier admin autenticado puede sobre **cualquier otro** admin:

- `PATCH { password }` → cambiar su contraseña (sin pedir la contraseña actual ni re-auth).
- `PATCH { reset_totp: true }` → resetear su 2FA (`totp_enabled = false` + nuevo secreto).

Encadenado: Admin A fija un password conocido a Admin B + resetea su TOTP → A puede loguearse como B y enrolar un TOTP nuevo = **takeover completo de cualquier par**. Para un deploy multi-admin es escalamiento lateral. Aceptable solo si todos los admins son de plena confianza mutua — pero conviene documentarlo como decisión explícita, o añadir: re-auth (password actual) para cambiar password propio, y separación de roles para tocar a terceros.

### 🟡 L1 — Sin revocación de sesión server-side

La sesión es un token HMAC stateless (`lib/admin-session.ts`). `requireAdmin` revalida contra DB pero solo chequea existencia + `totp_enabled`. Consecuencias:
- `logout` (`/auth/logout`) solo borra la cookie del cliente; un token robado sigue válido hasta `exp` (8h).
- Cambiar la contraseña **no** invalida las sesiones activas (la propia `docs/ADMIN.md` lo confirma).

La única palanca de revocación real es rotar `ADMIN_SESSION_SECRET` (mata todas) o resetear el TOTP del admin. Considerar un `session_version`/`token_version` por admin incluido en el payload y verificado en `requireAdmin`.

### 🟡 L2 — `GET /users/[id]/totp-setup` re-expone el secreto tras enrolar

El `POST` correctamente rechaza si `totp_enabled` (400 "TOTP ya está activo"), pero el `GET` no tiene ese guard: un admin con sesión completa puede re-pedir su `totp_uri`/QR (= el secreto compartido) en cualquier momento. Aumenta la superficie de fuga del bearer credential. Añadir el mismo guard `if (totp_enabled) return 400` al GET.

### 🟡 L4 — Validación incompleta en writes admin

`app/api/admin/data/promotions/route.ts` valida `discount` (1-100), `modality` (enum) y requeridos, pero **no**: que `card_types` ⊆ {credit,debit}, que `cap`/`min_purchase` sean números ≥0, que `days_of_week` esté en 0-6, ni que `start_date`/`end_date`/`verified_at` sean fechas válidas o que `end_date ≥ start_date`. Input malformado revienta en el `CHECK`/cast de Postgres → **500 en vez de 400**. Además `PATCH /merchants/[id]` (y similares) corren varios `UPDATE` separados sin transacción → fallo a mitad deja estado parcial. Los campos de texto (`name`, `source`, `conditions`) no tienen límite de largo. Severidad baja por ser admin-only autenticado, pero conviene validar en capa app.

### 🟡 L5 — Tablas de seguridad crecen sin poda

`admin_login_attempts` acumula una fila por intento fallido indefinidamente; `admin_audit_log` se consulta a 30 días pero nunca se purga. Agregar un job de limpieza (o `DELETE` por TTL) evita crecimiento monótono.

### Verificado y correcto (sin hallazgos) en el panel

- **bcrypt** cost 12 + comparación anti-enumeración (fake hash de tiempo constante).
- **TOTP** AES-256-GCM en reposo; clave vía scrypt desde secreto de app; window ±1.
- **Verificación HMAC** con `crypto.subtle.verify` (tiempo constante); `exp` chequeado.
- **`requireAdmin`** re-consulta DB en cada request → fail-closed (admin borrado/2FA reseteado pierde acceso al instante).
- **Cookie** `HttpOnly; Secure(prod); SameSite=Strict; Path=/` → SameSite=Strict es la defensa CSRF efectiva para las mutaciones.
- **`server-only`** + `node:crypto` confinan auth/cifrado al servidor.
- **Sin secretos** en working tree (`.env.example` solo placeholders); `.gitignore` cubre `.env*`.

## Bugs y correctness (núcleo público + cliente)

### Confirmado limpio
- **SQL:** todas las queries parametrizadas (tagged templates Neon). Sin inyección. Escape LIKE presente.
- **`lib/recommendations.ts`:** `calculateSavings` correcto (cap, min_purchase, amount≤0).
- **`/api/recommendations`:** valida `cardIds` (≤100 + `areValidIds`), `date` (regex + `Date` válida), `merchantId`. Día calendario robusto vía UTC. TZ Chile correcto.

### Observaciones
- **`calculateStackedSavings` (`lib/recommendations.ts`):** aplica descuentos sucesivos sobre el monto **ya reducido**, y evalúa `min_purchase` contra ese monto reducido — probablemente incorrecto: el mínimo de compra real se mide sobre el monto original de la transacción, no sobre el remanente. Revisar si el apilado de promos es siquiera un caso válido del dominio (las promos bancarias chilenas suelen ser excluyentes). Si no se usa, es código muerto.
- **`lib/types.ts`:** sigue siendo **código muerto** (ningún import lo referencia; todo usa los tipos `Api*` de `api-client.ts`). Heredado del audit 2026-06-10, aún abierto.
- **`public/sw.js`:** `console.info/error` sin guard de entorno llegan a la consola del usuario final (heredado). `isStaticAsset` matchea cualquier path `*.js`/`*.css`.

### Resueltos desde el audit 2026-06-10 (verificado en el código actual)
Cliente Neon ahora cacheado (`cachedClient`) · `buildUrl` usa base dummy (no crashea en SSR) · SW cache mismatch corregido (usa `STATIC_CACHE_NAME` consistente) · `useWallet` usa updater funcional (sin stale closure) · `/api/stats` con `::int` · regex de fecha + validación lógica · `SELECT *`/`p.*` → columnas explícitas · SW con guard `NODE_ENV==="production"` · `manifest.start_url` ahora `/app` · error boundaries (`app/error.tsx`, `app/global-error.tsx`).

## Documentación (¿está al día?)

Las docs están detalladas y con fecha 2026-06-13, pero hay inconsistencias con el código:

- **D1 🟠 (peligrosa) — `docs/ADMIN.md`, tabla "Seguridad del panel":** dice **Cookie `Path=/admin`** ("no se envía en requests a rutas no-admin"). El código setea `Path=/` (`lib/admin-session.ts`), y el commit `f075794 fix: session cookie path prevents API auth (login loop)` ya corrigió exactamente eso. La misma doc se contradice (la tabla "Cookie de sesión" dice `Path=/`). Si alguien "alinea" el código a `/admin`, **rompe la auth de `/api/admin/*`**. Corregir la fila a `Path=/`.
- **D2 — `docs/SECURITY.md`, sección "Cookies":** afirma *"`ow_standalone` — Única cookie de la aplicación"* y *"No se usan cookies de sesión, autenticación…"*. Falso desde el panel (`ow_admin_session`). Stale.
- **D3 — `docs/SECURITY.md`, "CSRF: ✅ No necesario — Solo endpoints GET… Sin cookies de sesión":** falso ahora (hay mutaciones admin + cookie de sesión). La defensa real es `SameSite=Strict`; debería decir eso.
- **D4 — `docs/SECURITY.md`, "Rate limiting ❌ Pendiente / La API no tiene rate limiting":** el API **admin** sí tiene. Aclarar público vs admin.
- **D5 — `docs/ADMIN.md`, API Reference:** formas de respuesta desactualizadas: `DELETE` devuelve `{status:"ok"}` (no `{ok:true}`); self-delete y last-admin devuelven **400** (la doc dice **403**); `PATCH {reset_totp}` devuelve `{status:"ok"}` **sin** QR (la doc muestra `{qr_data_url,…}`); `POST /users` no incluye `totp_enabled` en la respuesta.
- **D6 — `docs/ADMIN.md`, flujo de datos:** dice que los handlers usan `getAdminFromRequest()`; en realidad usan `requireAdmin()` (más fuerte, con check a DB). Subvende la seguridad real.
- **D7 — Reportes en `OptiWallet/`:** `audit-report.md` (2026-06-10) y `security-audit-2026-06-11.md` son **anteriores al panel admin** y la segunda afirma "sin autenticación". `docs/SECURITY.md` los enlaza como la referencia de auditoría → apunta a material que no cubre la superficie admin. Marcar como históricos y reemplazar por este reporte.
- **D8 — OpenAPI (`lib/openapi.ts`), schema `Id`:** patrón `^[a-zA-Z0-9_-]{1,64}$` omite el punto (`.`) que `lib/validate.ts` sí permite (`/^[A-Za-z0-9_.-]{1,64}$/`). Menor.

## Recomendaciones priorizadas

1. **Excluir `/api/admin/*` y `/admin` del Service Worker** (M1) — una línea, alto impacto.
2. **IP confiable para rate limiting** (M2) — `x-real-ip` en vez del primer XFF.
3. **Corregir `docs/ADMIN.md` cookie `Path=/admin` → `Path=/`** (D1) — riesgo de regresión de auth.
4. **Actualizar `docs/SECURITY.md`** (cookies, CSRF, rate limiting) y marcar los reportes en `OptiWallet/` como históricos (D2–D7).
5. **Decidir el modelo de privilegios admin** (M3): re-auth para cambio de password propio, y/o roles para operar sobre terceros.
6. Revocación de sesión (L1), guard en GET totp-setup (L2), validación de writes admin (L4), poda de tablas (L5).
7. `npm audit fix` (dev), endurecer `apply-schema.ts` (L3), limpiar `lib/types.ts`.

| M2 | IP spoofeable para rate limit | `lib/admin-guard.ts` `clientIp()`: prioriza `x-real-ip`, fallback al último hop de XFF |
| M3 | Takeover lateral entre admins | `PATCH /users/[id]` exige `current_password` del admin que ejecuta (re-auth), rate-limited; UI (`app/admin/users/[id]/page.tsx`) pide la contraseña actual en ambos formularios |
| L1 | Sin revocación de sesión | Columna `token_version` (schema), embebida en el token al firmar; `requireAdmin` la verifica (self-healing pre-migración); se incrementa en cambio de password y logout |
| L2 | GET totp-setup re-expone secreto | `GET /users/[id]/totp-setup` devuelve 400 si `totp_enabled` |
| L4 | Validación incompleta + PATCH no atómico | Validadores nuevos en `lib/validate.ts` (con tests); todos los PATCH (banks/cards/categories/merchants/promotions) hacen un único `UPDATE` atómico vía read-merge-write |

| I4 | Vulns dev transitivas | `npm audit fix` (árbol de producción a 0 vulns) |

**Paso operativo requerido antes/junto al deploy:** correr `npm run db:schema` para agregar la columna `token_version` (idempotente, `ADD COLUMN IF NOT EXISTS`, no destruye datos). El código es *self-healing*: funciona aunque la migración aún no se haya corrido (la revocación de sesión simplemente queda inactiva hasta entonces), así que no hay riesgo de lockout por orden de deploy.

**Fuera de alcance (infra/ops, no código):** I1 rate limiting de la API pública (Vercel WAF), I2 CSP con nonces (trade-off intencional documentado — rompería el static optimization de la landing), I3 rol de DB de solo lectura.

**Verificación:** la lógica de validación nueva se ejecutó de forma aislada (type-stripping de Node + asserts) → **OK**. El typecheck/test/lint completos no pudieron correrse end-to-end dentro del sandbox por una divergencia del mount de archivos; **correr localmente** `npm install` (restaura devDeps), `npx tsc --noEmit`, `npm test` y `npm run lint` antes de mergear.
